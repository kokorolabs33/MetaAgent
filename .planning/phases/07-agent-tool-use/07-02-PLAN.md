**Strategy for tool call SSE events:**

The tool calling loop runs inside `cmd/openaiagent/` (a separate binary). The executor in `cmd/server/` communicates with agents via A2A protocol (HTTP polling). The challenge: how do tool call events get from the agent binary to the executor's SSE broker?

**Approach: Tool call metadata in A2A status message + executor-initiated SSE events.**

The A2A protocol supports `state: "working"` with a `status.message` field for intermediate progress. When the executor polls (`tasks/get`), it can read the status message.

**B. `internal/executor/executor.go` — Detect tool call progress during polling.**

In the `pollUntilTerminal` method, after each `tasks/get` response with `state == "working"`, check if the status message contains tool call progress. Parse the message text for `tool_call_started:` or `tool_call_completed:` prefixes.

Add a helper method:

```go
func (e *DAGExecutor) checkToolProgress(ctx context.Context, taskID, subtaskID, agentID string, result *a2a.SendResult) {
    if result.State != "working" || result.Message == "" {
        return
    }
    msg := result.Message

    if strings.HasPrefix(msg, "tool_call_started:") {
        parts := strings.SplitN(msg[len("tool_call_started:"):], ":", 2)
        if len(parts) == 2 {
            toolName := parts[0]
            toolArgs := parts[1]
            e.publishEvent(ctx, taskID, subtaskID, "tool.call_started", "agent", agentID, map[string]any{
                "tool_name": toolName,
                "args":      toolArgs,
            })
        }
    } else if strings.HasPrefix(msg, "tool_call_completed:") {
        parts := strings.SplitN(msg[len("tool_call_completed:"):], ":", 2)
        if len(parts) == 2 {
            toolName := parts[0]
            summary := parts[1]
            e.publishEvent(ctx, taskID, subtaskID, "tool.call_completed", "agent", agentID, map[string]any{
                "tool_name": toolName,
                "summary":   summary,
            })
        }
    }
}
```

Call `checkToolProgress` inside `pollUntilTerminal` after each poll iteration where the state is "working". Track which tool progress messages have already been published to avoid duplicate events — use a `lastToolProgress string` variable, only publish if `result.Message != lastToolProgress`.

```tsx
"use client";

import { Search, CheckCircle2, Loader2 } from "lucide-react";
import { cn } from "@/lib/utils";
import type { ToolCallEvent } from "@/lib/types";

interface ToolCallStatusProps {
  event: ToolCallEvent;
}

const toolIcons: Record<string, React.ComponentType<{ className?: string }>> = {
  web_search: Search,
};

function formatToolDisplay(toolName: string, args?: string): string {
  if (toolName === "web_search" && args) {
    try {
      const parsed = JSON.parse(args) as { query?: string };
      if (parsed.query) return `Searching: "${parsed.query}"`;
    } catch {
      // fall through
    }
  }
  return `Running: ${toolName.replace(/_/g, " ")}`;
}

export function ToolCallStatus({ event }: ToolCallStatusProps) {
  const Icon = toolIcons[event.tool_name] ?? Search;
  const isActive = event.status === "started";

  return (
    <div
      className={cn(
        "flex items-center gap-2 px-4 py-1.5 text-xs",
        isActive ? "text-blue-400" : "text-gray-500",
      )}
    >
      <div className="flex size-5 items-center justify-center">
        {isActive ? (
          <Loader2 className="size-3.5 animate-spin" />
        ) : (
          <CheckCircle2 className="size-3.5" />
        )}
      </div>
      <Icon className="size-3.5" />
      <span>
        {isActive
          ? formatToolDisplay(event.tool_name, event.args)
          : `${event.tool_name.replace(/_/g, " ")} complete`}
      </span>
    </div>
  );
}
```

|----------|-------------|
| Executor -> Agent (A2A poll) | Executor reads tool progress markers from agent status message |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Information Disclosure | ToolCallStatus.tsx | accept | Tool names and search queries are visible to the authenticated user who created the task. This is intended behavior — transparency is the feature. No PII in tool args (agent-generated search queries). |
| T-07-07 | Spoofing | pollUntilTerminal tool progress | mitigate | Tool progress markers use a fixed prefix format (`tool_call_started:`, `tool_call_completed:`) that cannot be spoofed by regular agent text content. The executor only parses messages from authenticated A2A agents. |
| T-07-08 | Denial of Service | SSE tool events | accept | Tool call events are low-volume (1-10 per subtask). The existing broker buffer (64) handles this easily. No additional DoS vector introduced. |
</threat_model>
