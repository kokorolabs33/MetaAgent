# TaskHub V2 — Agent Orchestration Platform Design Spec

> **One-liner:** A platform for unified orchestration, management, and governance of AI Agents — Kubernetes for Agents.

## Design Decisions

| Decision | Choice |
|----------|--------|
| Codebase | Full rewrite, reuse Go + Next.js stack |
| Platform role | Orchestrate & manage only — does NOT run agents |
| Built-in LLM | Only for orchestrator (task decomposition/routing) |
| Execution agents | All external, accessed via adapters |
| Task model | Hierarchical: Task → Orchestrator → N SubTasks → N Agents |
| Agent call mode | Async polling (MVP); callback & streaming later |
| Human-in-the-loop | Agent @mentions user in group chat; user @replies to continue |
| Frontend | Dashboard (DAG pipeline) + Group Chat (@mention interaction) |
| State management | DB-backed state machine (MVP); Temporal-ready interface |
| Auth | Better Auth + Google OAuth, org-scoped |
| Multi-tenancy | Organization-based isolation |
| Testing | Mock Agent (CI) + real LLM Agent (manual) |

### External Agent Protocol (Minimum Contract)

```

```

**Send input (optional):**
```

Request: { "message", "data"? }
```

### Adapter Config Format (http_poll)

```json
{
  "submit": {
    "method": "POST",
    "path": "/v1/analyze",
    "body_template": { "prompt": "{{instruction}}", "context": "{{input.context}}" },
    "job_id_path": "$.id"
  },
  "poll": {
    "method": "GET",
    "path": "/v1/jobs/{{job_id}}",
    "interval_seconds": 5,
    "status_path": "$.state",
    "status_map": { "processing": "running", "done": "completed", "error": "failed" },
    "result_path": "$.output",
    "error_path": "$.error_message",
    "messages_path": "$.logs[*].text",
    "progress_path": "$.progress"
  },
  "send_input": {
    "method": "POST",
    "path": "/v1/jobs/{{job_id}}/reply",
    "body_template": { "response": "{{message}}" }
  }
}
```

**Template variables:** `{{instruction}}`, `{{input.xxx}}`, `{{job_id}}`, `{{message}}`
- Missing variable → empty string (not error). `{{input.foo}}` when input is nil → `""`
- Nested paths supported: `{{input.nested.field}}` traverses JSON
- Values injected as-is (string → string, number → number in JSON)

**Response extraction:** Use `github.com/PaesslerAG/jsonpath` (RFC 9535 compliant).
- Scalar paths (`$.id`, `$.state`): return single value. Missing required path (`job_id_path`, `status_path`) → error. Missing optional path → nil.
- Array paths (`$.logs[*].text`): return `[]string`. Empty array if path not found.

**Credential encryption:** `auth_config` JSONB is encrypted at rest using AES-256-GCM with a server key from `TASKHUB_SECRET_KEY` env var. Encrypt on write (agent registration/update), decrypt on read (adapter layer before calling agent). Key rotation: re-encrypt all agent auth_configs when key changes.

## 3. Orchestrator & Execution Engine

### Core Interfaces

```go
type Orchestrator interface {
    Plan(ctx context.Context, task Task, agents []Agent) (*Plan, error)
    Replan(ctx context.Context, task Task, failedSubtask SubTask) (*Plan, error)
}

type TaskExecutor interface {
    Execute(ctx context.Context, task Task, plan Plan) error
    Cancel(ctx context.Context, taskID string) error
    Signal(ctx context.Context, taskID string, input UserInput) error
}

type AgentAdapter interface {
    Submit(ctx context.Context, agent Agent, input SubTaskInput) (JobHandle, error)
    Poll(ctx context.Context, agent Agent, handle JobHandle) (AgentStatus, error)
    SendInput(ctx context.Context, agent Agent, handle JobHandle, input UserInput) error
}
```

### DAG Execution

```
Execute(task, plan):
  1. Create all subtask records (status=pending)
  2. Publish task.running event
  3. DAG executor loop:
     a. Find subtasks where: depends_on all completed + status=pending
     b. Apply concurrency limits (global max=10, per-agent max=3)
     c. Start each ready subtask in its own goroutine:
        - adapter.Submit() → get job_id
        - Polling loop:
          - adapter.Poll() → check status
          - running → grab messages for chat, wait interval, poll again
          - needs_input → waiting_for_input, @user in chat, block for Signal()
          - completed → update subtask, check downstream dependencies
          - failed → retry or mark failed
     d. All completed → task.completed
     e. Final failure → optional Replan (max 2 times) or task.failed
```

Polling interval: exponential backoff 1s → 2s → 5s → 10s → 30s (capped).

### Upstream Output Injection

When subtask s3 depends on s1 and s2, the executor injects their outputs. Input is computed at execution time and persisted to `subtasks.input` before calling `adapter.Submit()`.

```go
func buildSubtaskInput(st SubTask, allSubtasks []SubTask, agents []Agent) map[string]any {
    input := map[string]any{"instruction": st.Instruction}
    // Use agent name as key (more readable than UUID in agent's instruction)
    for _, depID := range st.DependsOn {
        dep := findSubtask(depID, allSubtasks)
        agent := findAgent(dep.AgentID, agents)
        input["upstream_"+agent.Name] = dep.Output
    }
    return input
}
```

### Retry Strategy

Three layers:
1. **Adapter layer** — network retries (timeout, 5xx, connection_refused). Transparent to executor. Backoff: 1s, 2s, 4s, max 3 attempts.
2. **SubTask layer** — agent returns `failed`. Retry up to `max_attempts` with backoff. Optionally switch to another agent with same capability.
3. **Task layer** — `orchestrator.Replan()` after final subtask failure. Max 2 replans per task (tracked in `tasks.replan_count`, persisted to DB).

### Replan Behavior

When a subtask fails finally and has blocked dependents:

```
1. Increment tasks.replan_count atomically
2. If replan_count > 2 → task.failed, no more replans
3. Call orchestrator.Replan(task, failedSubtask)
4. LLM receives: original task, current plan state, which subtask failed and why
5. LLM returns a PARTIAL plan: replacement subtasks for the failed one + its dependents
6. Executor:
   a. Cancel/remove blocked subtasks that are being replaced
   b. Insert new subtasks from replan (status=pending)
   c. Resume DAG loop — new subtasks will be picked up
```

Replan does NOT regenerate the entire plan — only the failed subtask and its downstream dependents. Already-completed subtasks are preserved.

### Human-in-the-loop Flow

```
1. Agent returns status=needs_input
2. SubTask → waiting_for_input
3. Event: subtask.waiting_for_input { message, options }
4. Chat: "@user research-agent needs your input: ..."
5. User posts message @research-agent in chat
6. Platform parses @mention → Signal(taskID, UserInput)
7. adapter.SendInput(agent, job_handle, input)
8. SubTask → running, polling resumes
```

### Signal Mechanism

The executor pre-registers a signal channel **before** setting the subtask status to `waiting_for_input` in the DB. This eliminates the race where Signal() arrives between DB update and channel registration.

```go
type DAGExecutor struct {
    signals   sync.Map  // map[string]chan UserInput
    cancels   sync.Map  // map[string]context.CancelFunc (per task)
}

// Polling goroutine: register channel BEFORE updating DB status
func (e *DAGExecutor) enterWaitingForInput(subtaskID string) UserInput {
    ch := make(chan UserInput, 1)
    e.signals.Store(subtaskID, ch)    // 1. register channel first
    updateSubtaskStatus(subtaskID, "waiting_for_input")  // 2. then update DB
    // Now Signal() can find the channel even if it arrives immediately
    input := <-ch
    e.signals.Delete(subtaskID)
    return input
}

// Signal() called from HTTP handler:
func (e *DAGExecutor) Signal(ctx context.Context, taskID string, input UserInput) error {
    ch, ok := e.signals.Load(input.SubtaskID)
    if !ok {
        return ErrSubtaskNotWaiting
    }
    ch.(chan UserInput) <- input
    return nil
}
```

Order of operations: register channel → update DB → publish event → wait. Signal() can only be called after the HTTP handler sees `waiting_for_input` in DB, and by that time the channel is already registered.

### Cancellation

Each task gets a `context.WithCancel`. Cancel() calls the cancel function, which propagates to all polling goroutines via `ctx.Done()`.

```go
func (e *DAGExecutor) Cancel(ctx context.Context, taskID string) error {
    if cancel, ok := e.cancels.Load(taskID); ok {
        cancel.(context.CancelFunc)()
    }
    updateTaskStatus(taskID, "cancelled")
    // All running subtasks' goroutines detect ctx.Done() and exit
    return nil
}
```

### Blocked Propagation

When a subtask reaches final failure (exhausted retries, no replan triggered):

Simple rule: **any failed dependency blocks the subtask.** We use the conservative approach — if ANY upstream dependency is failed, the subtask is blocked. This is simpler to implement and reason about than the "all paths" optimistic check.

```
1. Mark subtask as "failed"
2. Find all subtasks where failed_subtask_id is in their depends_on array
3. Mark each as "blocked"
4. Recursively: find subtasks depending on newly-blocked subtasks → also "blocked"
```

This means if s3 depends on s1 and s2, and s1 fails, s3 is blocked even though s2 might succeed. This is correct because s3's instruction was designed assuming both s1 and s2 outputs are available.

### Crash Recovery

On server restart, the executor scans DB for in-progress work:

```
1. Find subtasks WHERE status = 'running' AND poll_job_id IS NOT NULL
   → Resume polling using EXISTING poll_job_id (do NOT re-Submit)
   → adapter.Poll(agent, existingJobHandle) to check current state

2. Find subtasks WHERE status = 'waiting_for_input'
   → Re-register signal channels, wait for user input

3. Find subtasks WHERE status = 'running' AND poll_job_id IS NULL
   → Server crashed between Submit and storing job_id
   → Mark as failed, let retry logic re-submit

4. Find tasks WHERE status = 'running'
   → Create fresh context.WithCancel and store in cancels sync.Map
   → Re-register signal channels for any waiting_for_input subtasks
   → Re-enter DAG executor loop for each
```

Critical: recovery uses `Poll()` with the stored `poll_job_id`, never re-calls `Submit()`. This prevents duplicate job creation on the external agent. Recovery also re-registers cancel contexts so that `Cancel()` works on recovered tasks.

### Budget Enforcement

Pre-Submit hook in the executor:

```go
func (e *DAGExecutor) checkBudget(ctx context.Context, orgID string) error {
    budget := getOrgBudget(orgID)
    if budget <= 0 { return nil }  // no budget set

    var spent float64
    err := db.QueryRow(ctx,
        "SELECT COALESCE(SUM(cost_usd), 0) FROM audit_logs WHERE org_id = $1 AND created_at >= date_trunc('month', NOW())",
        orgID).Scan(&spent)
    if err != nil {
        return fmt.Errorf("check budget: %w", err)  // fail closed: deny if can't verify budget
    }

    if spent >= budget {
        return ErrBudgetExceeded  // task.failed with error="budget_exceeded"
    }
    if spent >= budget * getAlertThreshold(orgID) {
        publishEvent(Event{Type: "budget.alert", Data: map[string]any{"spent": spent, "budget": budget}})
    }
    return nil
}
```

Called before each `adapter.Submit()` and each `orchestrator.Plan()` (LLM call).

## 4. Auth & RBAC

### Authentication

Better Auth + Google OAuth. Database-backed sessions (not JWT — simpler revocation).

### Middleware Chain

```
Request → CORS → Auth Middleware → Org Middleware → RBAC Middleware → Handler
```

**Auth Middleware:** Extract session cookie → validate against Better Auth session store → populate `ctx.User`. Reject 401 if invalid.

**Org Middleware:** Extract `org_id` from URL path → verify user is a member of this org → populate `ctx.Org` and `ctx.OrgRole`. Reject 403 if not a member.

**RBAC Middleware:** Per-route permission check. Each route declares its required permission level. The middleware compares `ctx.OrgRole` against the RBAC matrix below. For "Own only" rules, the handler itself checks `resource.created_by == ctx.User.ID`.

```go
// Route registration with RBAC
r.With(RequireRole("member")).Post("/tasks", taskH.Create)
r.With(RequireRole("viewer")).Get("/tasks", taskH.List)
r.With(RequireRole("admin")).Post("/agents", agentH.Create)
```

### SSE & Event Replay

The `events` table is the durable backlog. The SSE broker is an in-memory fanout on top:

```
New event → save to DB → fanout to in-memory subscribers → SSE push
```

On SSE connect (`GET /tasks/:id/events`):
1. If client sends `Last-Event-ID` header:
   - Look up the event by ID to get its `created_at`
   - Replay events where `(created_at, id) > (lastEvent.created_at, lastEvent.id)` (composite comparison prevents duplicates when timestamps collide)
2. If no header → replay ALL events for this task from DB (full catchup)
3. After replay, subscribe to in-memory fanout for live events
4. Each SSE event includes `id: <event_id>` field so the browser auto-sends `Last-Event-ID` on reconnect

This gives: durable history, browser reconnection for free (standard SSE spec), and no event loss.

```
}
```

- Default limit: 20, max: 100
- Cursor is a base64-encoded composite of `created_at|id` (opaque to client)
- Server decodes cursor to build `WHERE (created_at, id) < ($1, $2)` query
- Ordered by `created_at DESC, id DESC` (newest first, deterministic)
- Filter params: `status`, `created_by`, `agent_id` (for tasks)

### Tech Stack (unchanged)

Next.js 15, React 19, Tailwind CSS 4, Zustand, shadcn/ui, React Flow, Better Auth.

## 6. Testing Strategy

### Mock Agent

Standalone Go service (`cmd/mockagent/`) with behavior controlled by instruction keywords:

| Keyword | Behavior | Tests |
|---------|----------|-------|
| `echo:` | Immediate completion with echoed content | Happy path |
| `slow:N` | Wait N seconds | Polling, progress |
| `fail:` | Return failed | Error handling |
| `fail-then-succeed:N` | Fail N times then succeed | Retry logic |
| `ask:` | Return needs_input | Human-in-the-loop |
| `messages:N` | Produce N chat messages | Message streaming |
| `progress:` | Return incremental progress | Progress tracking |

### Test Suites

1. **Basic Task Flow** — single subtask happy path, progress, cancellation
2. **DAG Execution** — parallel subtasks, dependency chains, blocked propagation
3. **Retry** — retry-on-failure, max retries, replan fallback
4. **Human-in-the-loop** — input request, user reply, resume
5. **Multi-Agent** — upstream output injection, group chat message flow
6. **Auth & RBAC** — unauthenticated rejection, role enforcement, agent permissions
7. **Adapter** — HTTP poll mapping, native direct, auth headers
8. **Audit & Cost** — audit log creation, cost tracking, budget enforcement

### Test Infrastructure

- `internal/testutil/` — `TestEnv` struct bootstraps DB, mock agent, API server
- CI: mock agent only (no real API keys)
- Manual: `cmd/llmagent/` wraps Claude API as a real external agent
